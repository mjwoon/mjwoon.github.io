🔧트러블 슈팅 ① — 대량 LLM 호출의 rate limit
🔴
상황
전체 학생 대상으로 OpenAI API를 일괄 호출하자 rate limit에 걸려, 배치가 중간에 실패하면 이후 학생 평가가 통째로 날아갔습니다.
🟢
해결
호출을 Celery 태스크 큐로 분리하고 자동 재시도 로직을 넣어, 일시적 실패와 rate limit을 흡수해 배치가 끝까지 완료되도록 했습니다.
💡
알게 된 점
외부 API에 의존하는 작업은 "실패할 수 있다"를 전제로 설계해야 하며, 재시도·큐잉이 편의 기능이 아니라 신뢰성의 핵심임을 체감했습니다.
🔧트러블 슈팅 ② — 선수강 관계 매핑의 임계값 트레이드오프
🔴
상황
비정형 텍스트로 된 선수강 조건(예: "기초 통계학 지식이 필요")을 Neo4j 그래프의 REQUIRES 관계로 자동 매핑하려 했습니다. 초기 임계값을 0.7로 잡았더니 정밀도는 71%로 높았지만 재현율이 2%에 그쳐 그래프 전체가 거의 텅 비는 문제가 있었습니다.
🟢
해결
임계값만 내리지 않고 두 축을 함께 탐색했습니다.
조정결과
임계값 0.7 → 0.6 완화엣지 7개 → 66개
α(SBERT:TF-IDF) 그리드 서치SBERT 90% + TF-IDF 10%, 임계값 0.6에서 F1 최고(0.20)
의미 기반 매칭을 주로 쓰되 소량의 어휘 검증을 섞어 노이즈를 잡는 게 가장 균형 잡힌 지점이었습니다.

알게 된 점
Precision과 Recall은 트레이드오프 관계이고, "정확도를 높인다"는 목표가 상황에 따라 오히려 시스템을 망가뜨릴 수 있음을 데이터로 확인했습니다. 정답은 하나가 아니라 서비스가 무엇을 원하는지에 따라 최적점이 달라진다는 걸 배웠습니다.

Kalm

AI / ML
한국어 댓글을 정상 / 공격적 표현 / 혐오·차별로 분류하는 모델. 단순 키워드 매칭으로는 잡히지 않는 문맥적 유해 표현 탐지가 목표였습니다. APEACH·KOLD·Unsmile 등 성격이 다른 여러 오픈소스 데이터셋을 통합해 약 8만 건 규모의 데이터 풀을 구축했습니다.
PythonPyTorchKcELECTRAko-sroberta-multitaskLightGBMFocal LossSMOTE
내가 기여한 부분
  • 데이터셋 통합 및 클래스 불균형 진단 (정상 70% / 혐오 23% / 공격적 7%)
  • baseline(SRoBERTa 임베딩 + LightGBM)부터 KcELECTRA fine-tuning까지 단계적 모델 설계·비교
  • Focal Loss 직접 구현 및 불균형 대응 기법 적용
🔧트러블 슈팅 — 90% 정확도라는 착시를 넘어서
🔴
상황
초기 baseline이 전체 정확도 약 90%를 기록했습니다. 수치만 보면 훌륭했지만 클래스별로 뜯어보니 데이터가 많은 '정상'은 F1 0.95로 잘 잡는 반면 '공격적 표현'은 F1 0.38에 그쳤습니다. 90%는 다수 클래스를 잘 맞춰 생긴 착시였고, Macro F1은 0.69에 불과했습니다.
🟢
해결
평가 지표를 정확도에서 Macro F1로 바꾸고, KcELECTRA를 fine-tuning하면서 Cross Entropy 대신 Focal Loss를 직접 구현해 소수 클래스에 학습을 집중시켰습니다.
지표Baseline+ Focal Loss
Macro F10.690.79 ⬆
공격적 표현 F10.380.55 ⬆
전체 정확도0.900.90
소수 클래스 F1이 0.38 → 0.55 (약 14% 개선), 정확도는 유지하면서 실질적 분류 능력이 좋아졌습니다.
💡
알게 된 점 (한계 포함)
남은 한계를 데이터로 규명했습니다. 공격적 표현 F1이 더 오르지 않은 건 한국어 욕설이 문맥에 따라 의미가 갈리는 Label Noise 때문이었고, 이 지점부터는 모델 기법보다 데이터 정제가 더 본질적인 해결책임을 지표로 확인했습니다. 높은 정확도가 곧 좋은 모델을 뜻하지 않으며, 한계를 솔직히 진단하는 것도 엔지니어의 역량이라는 걸 배웠습니다.

Backend

JavaSpring BootSpring SecurityJWT(RTR)RedisPostgreSQLJPAS3DockerPlaywright
내가 기여한 부분
  • JWT 인증 기반 회원 관리·스터디 신청·공지 CRUD REST API 설계·구현 (JPA Repository 패턴)
  • S3 연동 파일 업로드, 인증·인가 보안 설계

🔧트러블 슈팅 — 토큰 리프레시(RTR) 동시성 문제
🔴
상황
보안을 위해 리프레시 토큰을 1회용으로 교체하는 RTR을 적용했는데, 페이지 진입 시 여러 API가 거의 동시에 호출되면서 같은 리프레시 토큰으로 동시에 갱신을 시도했습니다. 한 요청이 토큰을 회전시키면 직후 요청은 이미 폐기된 토큰을 들고 와 거부당해, 먼저 정상 발급된 세션이 무효화되고 유저가 로그아웃됐습니다.
🟢
해결 — 프론트 · 백엔드 2단 방어
프론트(Axios 인터셉터) — 401이 처음 발생하면 갱신 요청을 한 번만 보내고, 그사이 들어온 요청들은 큐에서 Promise를 공유하며 대기 후 일괄 재시도. 동시 갱신 요청 자체가 발생하지 않도록 했습니다.

백엔드(Redis Grace Period) — 토큰 회전 시 "옛 토큰 → 새 토큰" 매핑을 Redis에 10초간 캐싱. 이미 회전된 토큰으로 다시 온 요청은 거부하지 않고 캐시된 새 토큰을 그대로 반환해 늦은 요청도 정상 동작하게 했습니다.
💡
알게 된 점
동시성 문제는 한 계층에서만 막으면 빈틈이 생깁니다. 프론트에서 요청을 줄이고(예방) 백엔드에서 안전망을 두는(방어) 2단 설계로, 보안(RTR)과 사용자 경험(끊김 없는 세션)을 동시에 지킬 수 있었습니다.
